## Non-negotiables

- **Always choose the simpler approach. Do not overengineer.** Write the minimum code that solves the actual problem in front of you â€” nothing speculative. No abstractions, interfaces, factories, strategy patterns, generic wrappers, or "extensibility hooks" introduced for a single caller or a hypothetical future need. No configuration options that nobody asked for. No defensive error handling for cases that cannot occur. No premature splitting of a 40-line service into five files. If a feature can be expressed as a straight-line function or a single repository call, do that â€” do not wrap it in a class hierarchy, an event bus, or a pipeline. Before adding a layer (DI provider, custom decorator, new module, base class, helper file), justify it against the actual current requirement; if the justification is "we might need it later," remove it. When two solutions both work, ship the smaller one. If a senior engineer would call it overcomplicated, it is â€” rewrite it simpler.

- **Fix every ESLint issue. Never silence one with an inline disable** (`// eslint-disable-next-line`, `/* eslint-disable */`, file-level disables, or per-rule disables in `eslint.config.mjs`). If a rule fires, the code is wrong â€” change the code, not the rule. If you genuinely believe a rule is misconfigured for this project, surface it and discuss before changing config; do not paper over the warning at the call site.

- **Never use raw SQL queries for CRUD.** All create / read / update / delete operations MUST go through the TypeORM repository pattern (`Repository<T>` methods: `find`, `findOne`, `save`, `update`, `delete`, `remove`, etc.). Do not use `dataSource.query(...)`, `EntityManager.query(...)`, or hand-written SQL strings for CRUD. The only exception is **high-complexity report / analytics generation** that genuinely cannot be expressed via the repository API â€” in that case use the TypeORM **QueryBuilder** (or a CTE via QueryBuilder) instead of raw SQL strings. If you believe a case requires raw SQL, surface it and discuss before writing it; do not slip raw SQL in silently.

- **Never use manual `@nestjs/schedule` cron decorators.** Do not import `Cron`, `CronExpression`, `Interval`, or `Timeout` from `@nestjs/schedule`, and do not introduce `ScheduleModule.forRoot()` for new scheduled work. Every recurring / time-based job MUST be implemented as a **BullMQ repeatable job** (e.g. `queue.add(name, data, { repeat: { pattern: '...' } })` or `{ repeat: { every: ms } }`) processed by a BullMQ worker. This keeps scheduling distributed-safe, observable in Redis, and consistent with the rest of the queue infrastructure. If you find existing `@nestjs/schedule` usage, surface it for migration â€” do not add new ones alongside.

- **Always use proper, valid TypeScript types. `any` is banned.** Do not use the `any` type anywhere in this codebase â€” not in parameters, return types, generics, casts (`as any`), declarations, or `// @ts-ignore` / `// @ts-expect-error` workarounds. Do not use manual runtime "type checking" (`typeof x === 'string'`, `'field' in obj`, ad-hoc shape guards) as a substitute for declared types â€” types come from interfaces, type aliases, DTO classes, entity classes, generics, or `zod`/class-validator schemas with inferred types. If a value is genuinely unknown at a boundary (external API response, parsed JSON, untyped library), use `unknown` and narrow it with a real type guard or schema validator, never `any`. If a third-party library lacks types, write or import a proper `.d.ts` declaration. If you think `any` is unavoidable, stop and surface it â€” do not commit it.

- **Every controller request and response must have explicit DTO-backed Swagger documentation.** Do not leave Swagger to infer `{}` request or response bodies. Any endpoint with `@Body()` must use a named request DTO class with `@ApiProperty` / `@ApiPropertyOptional` metadata and concrete examples or enum/default values for every field. Every controller route must declare its normal API response using the shared envelope Swagger helpers (`ApiOkData` or `ApiOkPaginated`) with a named response DTO that documents the actual payload shape. Do not use inline request/response classes inside controllers. If an endpoint returns a simple success result, use a DTO such as `SuccessResponseDto` and return that value so the runtime response still follows the normal `{ data, error }` API envelope.

- **Every controller endpoint must declare its complete success and error response shape, and every controller using the global response envelope must wire up typed request and response DTOs.** For every `@Get`, `@Post`, `@Put`, `@Patch`, `@Delete`, and list endpoint, declare BOTH: (a) the success response â€” `ApiOkData(ResponseDto, { description })` for single-resource success, `ApiOkPaginated(ItemDto, { description })` for list / paginated success, or `@ApiResponse({ status: 204, description })` for empty mutations (never an inline `schema: { ... }` object, never a missing success declaration); and (b) every error status the endpoint can actually emit via `ApiErrors(...statuses)` from `src/common/http/swagger.ts`, drawing from the documented set (`400`, `401`, `403`, `404`, `409`, `410`, `422`, `429`, `500`, `503`). List the statuses the handler can really produce â€” not a copy-pasted superset, not a missing subset. Every request body / query / param must be a typed DTO class with `@ApiProperty` (or `@ApiPropertyOptional`) on **every** field, including nested objects and array element types, so the kubb client emits real types instead of `unknown` / `any`. Every response shape must be a named DTO class (e.g. under `dto/responses/`) with `@ApiProperty` on every field â€” no inline object literals in `@ApiResponse({ schema: ... })`, no returning an untyped service result through the envelope. Endpoints that intentionally bypass the global envelope must use `@RawResponse()` from `src/common/http/raw-response.decorator.ts` AND document the raw schema explicitly with `@ApiResponse({ schema })`. This is a broken-window rule: when you touch a controller, every endpoint in that file must satisfy this contract before you commit â€” do not leave neighbouring endpoints with missing `ApiErrors`, missing `ApiOkData` / `ApiOkPaginated`, inline schemas, or untyped DTOs.
